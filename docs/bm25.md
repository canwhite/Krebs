# BM25 算法详解

## 1. 背景：为什么需要 BM25

在信息检索中，简单匹配词频（TF）有严重缺陷：文档越长命中次数越多，"的"字在所有中文文档中都高频出现却无意义。

BM25（Best Matching 25）是 **Okapi BM25** 的改进版本，由微软研究院在 1994 年提出，至今仍是搜索引擎和 RAG 系统的标准检索算法。它的核心思想：**结合词频（TF）与逆文档频率（IDF），同时对文档长度做归一化**。

## 2. 两个核心概念

### 2.1 IDF — 逆文档频率

一个词在越多文档中出现，说明它越普遍、越没有区分度。IDF 量化这个"区分度"。

```
IDF = log((N - d + 0.5) / (d + 0.5) + 1)
```
当然这个公式可以简化，简化后是：
```
IDF = log((N+1)/(d+0.5))
```
然后就很好理解了：
```
首先log可以表示信息量
这个词非常常见（比如“的”、“是”、“今天”）它几乎出现在每一篇文档中，所以 d 趋近于 N
log(1) 接近于0
这个值非常罕见的话，就会是一个正数且比较大
```

| 符号 | 含义 |
|------|------|
| `N` | 总文档数 |
| `d` | 包含该词的文档数 |
| `+0.5` | 拉普拉斯平滑，避免 d=N 时 IDF=0 |
| `+1` | 避免 d=0 时 log(0) |

**直观理解**：
- 词出现在所有文档中（d=N）→ IDF ≈ log(1) = 0，无信息量
- 词只出现在 1 个文档中（d=1）→ IDF ≈ log(N)，非常稀有，高信息量

### 2.2 TF Normalization — 词频归一化

简单词频（词在文档中出现次数）不能直接用，因为：
- 长文档天然比短文档有更多词频
- 词频增长不会无限线性

BM25 使用饱和函数控制词频增长：

```
TF_norm = (tf × (k1 + 1)) / (tf + k1 × (1 - b + b × docLen / avgDocLen))
```
我们需要理解这个公式：
```
上边是词频乘，下边是加

分子（乘法）决定“天花板有多高”；分母（加法）决定“爬上去有多费劲”。
---
为什么分子必须用“乘法”？（设定最高上限）
--
因为是乘法，它确立了得分的极限值。当 tf 趋近于无穷大时，分子比分母大，
但整个分数的极限是(k1+1)（约等于 2.2）。
--这里很反直觉，以为乘就是放大，乘也可以是限制，因为可以乘小的数

---
为什么分母必须用“加法”？（控制“饱和速度”）
---
这是整个公式最天才的地方。如果分母也是乘法，就会把tf约掉，只剩下k1+1 / K
tf 就被约掉了，那就完全看不出“出现次数多”的优势了。
正因为分母是加法，它才创造出了一个极其美妙的“边际效应递减”曲线。我们用数学直觉来看两种情况：

后边的那一串可以理解为长度惩罚，
对于短文档：
K 很小。分母就是 tf+小。这意味着你的词频 tf 的每一次增加，都能显著拉大分母，让得分快速上升。


对于长文档：
K 很大。分母就是 tf+大。这就导致前期的多次增加被这个巨大的加数 K 给“吞没”了。
只有当你把词频 tf 积累到非常大，大到超过 K 时，得分才开始明显增长。

```

| 符号 | 含义 | 本项目值 |
|------|------|----------|
| `tf` | 词在当前文档中的原始出现次数 | - |
| `k1` | 词频饱和参数，控制增长速率 | **1.5** |
| `b` | 文档长度归一化参数 | **0.75** |
| `docLen` | 当前文档的 token 数 | - |
| `avgDocLen` | 所有文档的平均 token 数 | - |

**直观理解**：
- `k1=1.5` 意味着当 `tf=1.5` 时，`TF_norm ≈ 1`；`tf` 继续增大时，`TF_norm` 增长变慢，逐渐趋于饱和
- `b=0.75` 控制文档长度的影响：文档越长，归一化分母越大，词频被压得越低

**TF_norm 曲线（k1=1.5, avgDocLen 归一化）**：

```
tf=0  → TF_norm = 0/(1.5×1) = 0
tf=1  → TF_norm = 2.5/(2.5+1.5×0.5) ≈ 0.77
tf=2  → TF_norm = 4.5/(3.5+1.5×0.5) ≈ 0.90
tf=5  → TF_norm = 7.5/(6.5+1.5×0.5) ≈ 0.96
tf=∞  → TF_norm → 1.0 (饱和上限)
```

## 3. BM25 最终公式

对于一个查询 Q（多个词 q1, q2, ...）和一个文档 D：

```
BM25(D, Q) = Σ_{qi∈Q} IDF(qi) × (tf(qi,D) × (k1 + 1)) / (tf(qi,D) + k1 × (1 - b + b × |D| / avgDL))
```
如果我们简化就很好理解了,直接乘出来的：
```
Score(term)=IDF(term)×TF_norm(term)
```

多词查询时，对每个查询词分别计算 BM25 得分，然后求和。

## 4. 本项目的 BM25 实现

### 4.1 代码映射

```typescript
export const K1 = 1.5;
export const B = 0.75;

export function bm25Search(
  index: SearchIndex,
  queryTokens: string[],
  excludeSessionId: string,
  topK: number = 5
): SessionIndexEntry[] {
  const N = index.sessionCount;

  // Step 1: 计算每个 query token 的 IDF
  const df = new Map<string, number>();
  for (const qt of queryTokens) {
    let count = 0;
    for (const entry of index.entries.values()) {
      if (entry.firstQuestionTokens.includes(qt)) count++;
    }
    df.set(qt, count);
  }

  // Step 2: 对每个 session 计算 BM25 score
  const scores = new Map<string, number>();
  for (const [sessionId, entry] of index.entries) {
    if (sessionId === excludeSessionId) continue;

    let score = 0;
    for (const qt of queryTokens) {
      const tf = entry.firstQuestionTokens.filter(t => t === qt).length;
      if (tf === 0) continue;

      // IDF
      const d = df.get(qt) ?? 0;
      const idf = Math.log((N - d + 0.5) / (d + 0.5) + 1);

      // TF Normalization
      const docLen = entry.tokenCount;
      const avgDocLen = index.avgTokenCount || 1;
      const tfNorm = (tf * (K1 + 1)) / (tf + K1 * (1 - B + B * docLen / avgDocLen));

      score += idf * tfNorm;
    }

    // Tiebreaker: createdAt 升序（越新越好），极小权重避免不稳定
    scores.set(sessionId, score + entry.createdAt / 1e15);
  }

  // Step 3: 排序取 topK
  return [...scores.entries()]
    .sort((a, b) => b[1] - a[1])
    .slice(0, topK);
}
```

### 4.2 流程图

```
查询: "如何配置 WASM sandbox"
     │
     ▼
queryTokens = ["如何", "配置", "WASM", "sandbox"]
     │
     ▼
DF 计算 ─────────────────────────────────────┐
│ q="如何": 出现在 5 个 session 中 → d=5      │
│ q="配置": 出现在 12 个 session 中 → d=12   │
│ q="WASM": 出现在 3 个 session 中 → d=3     │
│ q="sandbox": 出现在 8 个 session 中 → d=8  │
└────────────────────────────────────────────┘
     │
     ▼
对每个 session (排除当前 session) 计算 score
     │
     ├── Session A: ["如何", "配置", "WASM", "sandbox"] 全中
     │   TF("WASM")=1, docLen=10, avgDL=8
     │   TF_norm ≈ 1.38
     │   IDF("WASM") ≈ log((N-3+0.5)/(3+0.5)+1)
     │   score_A = Σ IDF × TF_norm ≈ 8.5
     │
     ├── Session B: ["配置", "sandbox"] 中 2 个
     │   score_B ≈ 4.2
     │
     └── Session C: 无命中 → score = 0
     │
     ▼
排序 + Tiebreaker: score 相同则 createdAt 大的排前
     │
     ▼
返回 top-5 SessionIndexEntry（带 score 字段）
```

### 4.3 Tiebreaker 的必要性

BM25 可能出现不同文档 score 完全相同的情况（如查询词都未命中，score 均为 0）。此时用 `createdAt / 1e15` 作为极小权重叠加：

```typescript
scores.set(sessionId, score + entry.createdAt / 1e15);
```

- `1e15` 足够大，确保不会影响正常的 score 排序
- `createdAt` 数值本身是毫秒级时间戳，范围在 `1e12` 左右
- 新 session 的 createdAt 更大，排在前面

## 5. 短查询 Fallback

当查询 token 少于 2 个时，BM25 效果差（IDF 无法有效区分）。此时 fallback 为**按时间倒序返回最近 2 个 session**：

```typescript
if (queryTokens.length < 2) {
  return [...index.entries.values()]
    .filter(e => e.sessionId !== excludeSessionId)
    .sort((a, b) => b.createdAt - a.createdAt)
    .slice(0, 2);
}
```

## 6. 中文分词策略

### 6.1 为什么需要分词

BM25 基于词（token）匹配。中文没有天然空格分隔，需要分词。

### 6.2 两种策略

| 策略 | 适用场景 | 精度 |
|------|----------|------|
| nodejieba | 安装了 nodejieba | 高（语义分词） |
| charNgram | nodejieba 不可用 | 中（字符 n-gram） |

**jieba 优先**（异步）：
```typescript
export async function tokenizeForIndex(text: string): Promise<string[]> {
  if (hasChinese(text)) {
    const jiebaResult = await tryJieba(text);
    return (jiebaResult ?? charNgram(text, 2, 4))
      .filter(t => !STOP_WORDS.has(t) && t.length > 1);
  }
  return text.split(/\s+/).filter(t => !STOP_WORDS.has(t) && t.length > 1);
}
```

**charNgram fallback**（同步，用于查询时）：
```typescript
export function charNgram(text: string, min = 2, max = 4): string[] {
  const tokens: string[] = [];
  for (let i = 0; i < text.length; i++) {
    for (let len = min; len <= max && i + len <= text.length; len++) {
      tokens.push(text.slice(i, i + len));
    }
  }
  return [...new Set(tokens)]; // 去重
}
```

例如"配置WASM"经过 charNgram(2,4) 得到：`["配置", "配W", "WAS", "WASM", "ASM", "sandbox", "and", ...]`

### 6.3 停用词过滤

```typescript
export const STOP_WORDS = new Set([
  '的', '了', '在', '是', '我', '有', '和', '就', '不', '人',
  '都', '一', '一个', '上', '也', '很', '到', '说', '要', '去',
  '你', '会', '着', '没有', '看', '好', '自己', '这', '怎么',
  '什么', 'the', 'a', 'an', 'is', 'are', 'was', 'were',
]);
```

停用词在 IDF 计算中贡献极小但会干扰结果，因此从索引和查询中都过滤掉。

## 7. 索引结构

```typescript
interface SearchIndex {
  version: number;           // schema 版本，用于 cache bust
  builtAt: number;           // 索引构建时间，用于 TTL
  sessionCount: number;      // session 总数，用于 IDF 计算
  avgTokenCount: number;     // 平均 token 数，用于 TF 归一化
  entries: Map<string, SessionIndexEntry>;
}

interface SessionIndexEntry {
  sessionId: string;
  filePath: string;
  firstQuestion: string;      // 分词前的原始问题（用于展示）
  firstQuestionTokens: string[];  // 分词后的 token 数组（用于检索）
  tokenCount: number;        // token 数量，用于 TF 归一化
  createdAt: number;         // 用于 tiebreaker
  score?: number;            // BM25 score（附加到条目上，用于去重）
}
```

## 8. 缓存策略

```typescript
const INDEX_VERSION = 1;
const CACHE_TTL_MS = 24 * 60 * 60 * 1000; // 24h

function isCacheValid(index: SearchIndex | null): boolean {
  if (!index) return false;
  if (index.version !== INDEX_VERSION) return false;           // 版本变化
  if (index.sessionCount !== getAllSessions().length) return false; // session 数量变化
  if (Date.now() - index.builtAt > CACHE_TTL_MS) return false;     // TTL 过期
  return true;
}
```

缓存失效条件（三者之一触发）：
1. **版本变化**：`INDEX_VERSION` 递增时强制重建
2. **session 数量变化**：新增或删除 session 时需要重建
3. **TTL 过期**：24 小时自动失效

## 9. 参数选择依据

| 参数 | 值 | 选择依据 |
|------|-----|----------|
| `K1` | 1.5 | BM25 经典值(1.2-2.0)，词频饱和速度适中 |
| `B` | 0.75 | BM25 经典值(0.5-0.8)，文档长度影响较大但不过度惩罚长文档 |
| `TOP_K` | 5 | 初筛数量，balance 召回率与性能 |
| `TOP_K_FINAL` | 2 | 最终注入数量，避免 context 过长 |
| `MAX_CHARS_PER_SESSION` | 1000 | 每个 session 提取上限，避免单条过长 |

## 10. 一个完整例子

**场景**：用户问"How to configure WASM sandbox"

**Step 1**：`preprocessForQuery` 预处理
```
输入: "How to configure WASM sandbox"
NFC:  "How to configure WASM sandbox"
过滤: "How to configure WASM sandbox"
分词: ["how", "configure", "was", "wasm", "sandbox"]
过滤停用词: ["configure", "wasm", "sandbox"]  ("how"是停用词)
```

**Step 2**：假设索引中有 100 个 session（`N=100`）

假设 "configure" 出现在 20 个 session（`d=20`）：
```
IDF("configure") = log((100-20+0.5)/(20+0.5)+1) = log(80.5/20.5+1) = log(4.93) ≈ 1.59
```

假设 "wasm" 出现在 5 个 session（`d=5`）：
```
IDF("wasm") = log((100-5+0.5)/(5+0.5)+1) = log(95.5/5.5+1) = log(18.36) ≈ 2.91
```

**Step 3**：计算 Session #42 的 BM25 分数

假设 Session #42 的 `firstQuestionTokens` = ["how", "configure", "wasm", "sandbox", "security"]

- `tf("configure") = 1`, `tf("wasm") = 1`
- `docLen = 5`, `avgDocLen = 6`

```
TF_norm("configure") = (1 × 2.5) / (1 + 1.5 × (1 - 0.75 + 0.75 × 5/6))
                     = 2.5 / (1 + 1.5 × 0.375)
                     = 2.5 / 1.5625 ≈ 1.60

TF_norm("wasm") = 同样计算 ≈ 1.60

BM25("configure") = 1.59 × 1.60 ≈ 2.54
BM25("wasm") = 2.91 × 1.60 ≈ 4.66
BM25("sandbox") = ... (类似计算)

Total Score ≈ 8.5
```

**Step 4**：按 score 排序，取 top-2 返回
