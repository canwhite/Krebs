## 1. `lib/session-repository.ts` — Interface exists, no adapters

**Files**: `lib/session-repository.ts` (59 lines), `server/session-service.ts`

## 3. `server/handlers/` — Thin wrappers with 1:1 interface:impl ratio

**Files**: `server/handlers/AuthHandler.ts`, `server/handlers/StopHandler.ts`, `server/handlers/index.ts`, `server/ws-router.ts`, `server/routes/index.ts`

**Problem**: `AuthHandler` (20 lines) and `StopHandler` (24 lines) are pass-through modules — their interface is the same complexity as their implementation. `ws-router.ts` could inline these as plain handler objects. `routes/index.ts` (7 lines, just exports `corsHeaders`) is the same pattern.

**Solution**: Inline these handlers into `ws-router.ts` as a `Record<MessageType, Handler>` map. The dispatch table replaces the handler class objects. Move `corsHeaders` into each route file that uses it (it's only used in 2 places).

**Benefits**: Eliminates indirection that provides no leverage. Locality improves — one file shows all routing logic.

## 4. `server/event-subscription.ts` — Monolithic event router

**Files**: `server/event-subscription.ts` (258 lines)

**Problem**: One function handles `message_start`, `message_end`, `message_update` (with 8 sub-types), `tool_execution_start/end`, `turn_end`, `agent_end` — and parses think tags inline. Adding one new event type requires modifying this monolith.

**Solution**: Extract into composable event handlers: `MessageEventHandler`, `ToolEventHandler`, `ThinkTagParser`, `SessionEndHandler`. The subscribe function composes them.

**Benefits**: Locality per event type. Each handler can be understood, tested, and modified independently. Tests survive because the interface (subscribe function) stays the same. Leverage: the same `subscribe()` interface works for all composed handlers.

### 4. `event-subscription.ts` 拆分 — 保持事件顺序是最高优先级

**风险**: 拆分后 `message_end` 和 `turn_end` 的顺序改变会破坏状态机。

**拆分原则**:
- **不改变事件分发顺序** — 这是硬约束
- 每个 handler 只负责"怎么处理这个事件"，不负责"什么时候发"
- 顺序由顶层 `subscribe` 函数的 `if/else` 链控制

**具体操作**:
```typescript
// 拆分后的 compose 结构
function subscribe(ws, session, logger, options) {
  const messageHandler = new MessageEventHandler(ws, logger, options);
  const toolHandler = new ToolEventHandler(ws, logger);
  const sessionEndHandler = new SessionEndHandler(ws, logger);

  return session.subscribe((event) => {
    if (event.type === "message_start")  return messageHandler.onMessageStart(event);
    if (event.type === "message_end")    return messageHandler.onMessageEnd(event);  // 顺序保留
    if (event.type === "message_update") return messageHandler.onMessageUpdate(event);
    if (event.type === "tool_execution_start") return toolHandler.onStart(event);
    if (event.type === "tool_execution_end")   return toolHandler.onEnd(event);
    if (event.type === "turn_end")      return sessionEndHandler.onTurnEnd(event);
    if (event.type === "agent_end")     return sessionEndHandler.onAgentEnd(event);
    // 新事件类型 → 新增一行，不改现有顺序
  });
}
```

**测试要求**: 拆分后必须有端到端测试验证 `message_start → message_update → message_end → turn_end → agent_end` 的 WS 消息顺序与拆分前完全一致。

### 10. `session-transcript.ts` — `thinking` 类型处理不一致

**Files**: `lib/session-transcript.ts` lines 33–34 vs `server/event-subscription.ts` lines 73–85

**问题**:

`extractFromMessages`（HTTP API 路径，`messages.ts`）:
```typescript

.filter((c: any) => c.type === "text" || c.type === "thinking")
.map((c: any) => (c.type === "thinking" ? c.thinking : c.text))
```

### 11. `event-subscription.ts` — `isProcessingMessageEnd` 锁无实际保护

**File**: `server/event-subscription.ts` lines 24, 196–207

**问题**:
```typescript
let isProcessingMessageEnd = false;  // line 24

// line 196-207
if (event.type === "message_end") {
  if (isProcessingMessageEnd) {
    logger.log("[SESSION] message_end 已在处理中，跳过重复事件");
    return;
  }
  isProcessingMessageEnd = true;
  logger.log("[SESSION] message_end 收到，等待 turn_end");
  isProcessingMessageEnd = false;  // ❌ 函数结束时立即重置，锁无效
}
```

这个锁的目的是防止并发处理同一个 `message_end` 事件，但 `isProcessingMessageEnd = false` 在 if 块末尾同步执行，锁在函数退出时已释放。如果两个 `message_end` 事件在同一个 event loop tick 到达（极短间隔），第二个仍会通过检查。

**Solution**:
如果这个锁的意图是"跳过快速连续的双重 `message_end`"，当前实现已经部分有效，但更好的方式是引入 Promise-based 状态：
```typescript
let messageEndPromise: Promise<void> | null = null;

// line 196-207 改为:
if (event.type === "message_end") {
  if (messageEndPromise) {
    logger.log("[SESSION] message_end 已在处理中，跳过");
    return;
  }
  messageEndPromise = (async () => {
    try {
      // 异步处理逻辑
    } finally {
      messageEndPromise = null;
    }
  })();
  return;
}
```

### 12. `think-parser.ts` — 废弃 API 的全球状态泄漏风险

**File**: `server/think-parser.ts` lines 111–132

**问题**:
```typescript
// line 111-112
const thinkParserStates = new Map<any, ThinkParser>();  // 全局 Map

function parseThinkTagsFromDelta(delta: string, ws: any): ParsedDelta {
  let parser = thinkParserStates.get(ws);  // 依赖 ws 对象引用作为 key
  if (!parser) {
    parser = new ThinkParser();
    thinkParserStates.set(ws, parser);  // ws 对象永不从 Map 删除（除非显式调用 cleanup）
  }
  return parser.parse(delta);
}
```

依赖 `ws` 对象引用作为 Map key。如果 `cleanupThinkParserState` 调用时机不对（WebSocket close 之前抛出异常），`ws` 对象会永远留在 Map 中。虽然当前代码有 try-catch 保护 close handler，但这个 global Map 设计仍然是脆弱的——任何未捕获的异常在 `parseThinkTagsFromDelta` 调用期间都会导致 leak。

**Solution**:
将 `parseThinkTagsFromDelta` 标记为 `@deprecated`，迁移所有调用方使用 `ThinkParser` 实例：

```typescript
// server/event-subscription.ts 中：
// 改为在连接建立时创建 ThinkParser 实例，存入 ws.data
(ws as any).data.thinkParser = new ThinkParser();

// 每次收到 delta 时：
const parser: ThinkParser = (ws as any).data.thinkParser;
```

### 13. `messages.ts` — HTTP API 和 WebSocket 的 `thinking` 可见性策略不统一

**Files**: `server/routes/messages.ts` line 99，`server/event-subscription.ts` line 90

**现状**:

HTTP API (`messages.ts` line 99) — `thinking` **可见**:
```typescript

```

WebSocket (`event-subscription.ts` line 90) — `thinking` **不可见**（只发送 `think_block` 事件，不合并到 `textDelta`）:
```typescript

// textDelta 是普通内容（不包含 thinking）
```

**分析**: 这是**有意设计**还是**疏漏**？

WebSocket 将 thinking 作为独立的 `think_block` 事件发送，前端可以单独渲染。HTTP API 将 thinking 内容合并到 `response.text` 中返回给调用方。两者策略不同：

- WebSocket 前端：thinking 独立显示
- HTTP API 调用方：thinking 合并在 response 中

如果前端期望 `response.text` 包含 thinking 内容（用于 TTS 或纯文本展示），这两个路径应该统一。

**Solution**:
明确产品策略后选择其一：

| 策略 | HTTP | WebSocket |
|------|------|-----------|
| A（thinking 合并）| `response.text` 包含 thinking | `turn_end.content` 包含 thinking |
| B（thinking 分离）| `response.thinking` 单独字段 | `think_block` 事件独立发送 |

建议采用策略 B（当前 WebSocket 路径），因为前端需要独立控制 thinking 的渲染位置。将 `extractFromMessages` 中对 `thinking` 的处理移除，恢复为只处理 `text` 类型。同时 `generatedContent` 字段只返回纯 text。

### 29. `lua-runtime.ts` — `AsyncMutex` 实现 bug

**File**: `tools/lua-runtime.ts` lines 24–54

**问题**:
```typescript
// line 24-54
class AsyncMutex {
  private mutex = Promise.resolve();

  async withLock<T>(fn: () => Promise<T>): Promise<T> {
    const release = this.mutex;  // 保存当前链

    let unlock: () => void;
    const lock = new Promise<void>((resolve) => { unlock = resolve; });

    // 将新锁添加到链中
    this.mutex = this.mutex.then(async () => {
      await lock;  // ❌ 等待 lock，但 lock 从未被 resolve
    });

    try {
      await release;  // 只等待旧链完成，不等待当前操作的 lock
      return await fn();
    } finally {
      unlock!();  // unlock 在 finally 中调用，但 this.mutex 的 then 从未等待这个 unlock
    }
  }
}
```

这个实现的问题：`this.mutex` 的 then 等待 `lock`，但 `lock` 只在 `finally` 中调用 `unlock()`。然而 `this.mutex` 的 then 是**异步**添加到链中的，它等待 `lock` 的 resolve 发生在 `finally` 之后（因为 `this.mutex = this.mutex.then(...)` 是同步赋值，then 的执行是异步的）。实际上这个 mutex 的行为是正确的（它确实串行化了），但代码逻辑混乱——`release` 只是用来确保顺序，真正的锁是 `this.mutex` 链。

然而更严重的问题：`mutex.withLock` 在 `luaRuntime.execute` 中每次调用。如果前一个任务还在执行中（`mutex` 还在等待），新的调用会等待整个链完成。这实际上是正确的互斥锁行为，但 `unlock()` 在 `finally` 中调用时，`this.mutex` 的 then 可能还没开始执行。

**实际 bug**: 如果 `fn()` 抛出异常，`unlock()` 仍会被调用（finally），这是对的。但 `this.mutex` 的 then 链中的 `await lock` 会永远等待，因为 unlock 只被调用一次。

**Solution**: 重写 mutex 实现：
```typescript
class AsyncMutex {
  private current: Promise<void> = Promise.resolve();

  async withLock<T>(fn: () => Promise<T>): Promise<T> {
    const release = this.current;
    let unlock: () => void;
    this.current = new Promise(resolve => { unlock = resolve; });

    try {
      await release;
      return await fn();
    } finally {
      unlock!();
    }
  }
}
```

```typescript
export async function waitForAgents(
  listAgentsFn: typeof listAgents,
  parentSessionId: string,
  agentIds: string[],
  timeoutMs: number = 30000
): Promise<Map<string, AgentResult>> {
  const deadline = Date.now() + timeoutMs;
```

```typescript
private getCronDelay(cron: string): number {
```

```typescript
  for (let dayOffset = 0; dayOffset < 7; dayOffset++) {
    for (let secondOffset = 0; secondOffset < 86400; secondOffset++) {
      const check = new Date(now.getTime() + dayOffset * 86400000 + secondOffset * 1000);
      if (this.matchesCronAt(cron, check)) {
        return dayOffset * 86400000 + secondOffset * 1000;
      }
    }
  }
  return Infinity;
}

// 或更高效的直接计算
private getCronDelay(cron: string): number {
  // 直接计算下一个匹配时间点
  const now = new Date();
  for (let min = 0; min < 10080; min++) { // 最多 7 天 * 1440 分钟
    const check = new Date(now.getTime() + min * 60000);
    if (this.isCronDueAt(cron, check)) {
      return min * 60000;
    }
  }
```

```typescript
} else {
  // append to existing
}
```

`ws.onmessage` 是异步的，在 React state 更新完成之前，如果 WS 消息快速连续到达，`streamingMessageIdRef.current` 仍为 `null`，会创建两个新消息而不是追加。

**Solution**:
使用 ref 锁定防止重复创建：
```typescript
const creatingMessageRef = useRef(false);

case "text_delta":
  if (!streamingMessageIdRef.current && !creatingMessageRef.current) {
    creatingMessageRef.current = true;
    const newId = crypto.randomUUID();
    streamingMessageIdRef.current = newId;
    setMessages((prev) => [...prev, { id: newId, ... }]);
    creatingMessageRef.current = false;
  } else if (streamingMessageIdRef.current) {
    // append
  }
  break;
```

### 67. `routes/sessions.ts` — 提取 content 时未检查 `msg.content?.[0]`

`msg.content` 可能是字符串、数组，或其他格式。如果数组为空或 `[0]` 没有 `text` 字段，返回空字符串。但更危险的是没有处理 `toolResult` 的 content 格式（它是 `{ type: "text", text: "..." }` 结构）。

**Solution**:
```typescript
function extractContent(msg: any): string {
  if (!msg.content) return "";
  if (typeof msg.content === "string") return msg.content;
  if (Array.isArray(msg.content)) {
    return msg.content
      .filter((c: any) => c.type === "text")
      .map((c: any) => c.text)
      .join("");
  }
  return "";
}
```
